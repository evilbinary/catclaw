# OpenClaw 设计理念、架构与核心原理总结

## 一、设计理念

OpenClaw 采用了以下核心设计理念：

1. **单网关多通道架构**：一个统一的网关服务管理所有聊天渠道（WhatsApp、Telegram、Discord 等），提供一致性体验
2. **嵌入式智能体**：使用 pi-mono 作为核心运行时，不依赖外部智能体进程
3. **工作区即记忆**：所有记忆、配置和工作文件都存储在工作区的 Markdown 文件中，易于备份和迁移
4. **会话隔离**：支持多智能体路由，每个智能体拥有独立的工作区、身份验证和会话
5. **插件化扩展**：通过插件系统支持额外的渠道、技能和功能
6. **单一事实来源**：TypeBox 模式作为网关协议的唯一真实来源，驱动代码生成、验证和文档

## 二、整体架构

### 1. 网关架构

- **核心组件**：单进程网关守护进程
- **通信方式**：WebSocket API，支持请求/响应模式和服务器推送事件
- **客户端连接**：
  - 控制面客户端（macOS 应用、CLI、WebUI）通过 WebSocket 连接
  - 节点（iOS/Android/headless）也通过 WebSocket 连接，提供设备功能
- **Canvas 服务**：网关同时提供 Canvas 界面和 A2UI 主机

### 2. 智能体运行时

- **单个嵌入式智能体**：基于 pi-mono 运行时
- **工作区**：单一工作目录（`~/.openclaw/workspace`）作为工具和上下文的唯一工作目录
- **启动文件**：AGENTS.md、SOUL.md、TOOLS.md、IDENTITY.md、USER.md、BOOTSTRAP.md 等在会话开始时注入上下文
- **技能加载**：从三个位置加载技能（工作区优先）
  - 内置技能（随安装包提供）
  - 管理/本地技能：`~/.openclaw/skills`
  - 工作区技能：`<workspace>/skills`

### 3. 多智能体路由

- **隔离性**：每个智能体拥有独立的工作区、agentDir、会话存储和身份验证配置
- **绑定机制**：通过 bindings 配置将入站消息路由到特定智能体
- **路由规则**：最具体匹配优先（peer 匹配 > guildId > teamId > accountId > 通道级 > 默认）
- **每智能体沙箱和工具**：支持每个智能体独立的沙箱和工具限制

## 三、核心原理

### 1. 上下文管理

- **系统提示**：OpenClaw 为每次智能体运行构建自定义系统提示，包含：
  - 工具列表 + 简短描述
  - 安全提醒
  - 技能列表（名称、描述、位置）
  - 工作区位置
  - 时间（时区信息）
  - 运行时元数据
  - 注入的工作区启动文件
- **启动文件注入**：默认注入 AGENTS.md、SOUL.md、TOOLS.md 等工作区文件，有大小限制
- **会话修剪**：每次 LLM 调用前从内存上下文中修剪旧的工具结果
- **压缩**：当会话接近或超过模型上下文窗口时，自动压缩旧对话并可能重试原始请求

### 2. 会话管理

- **会话键**：
  - 直接聊天默认为 `agent:<agentId>:main`
  - 群组/频道有独立的键
  - 支持 `dmScope` 配置：main、per-peer、per-channel-peer、per-account-channel-peer
- **安全 DM 模式**：多用户设置推荐使用 `per-channel-peer` 隔离上下文
- **会话重置**：每日重置（默认网关主机本地时间 4:00 AM）、空闲重置、手动重置（/new、/reset）
- **存储**：
  - 会话元数据：`~/.openclaw/agents/<agentId>/sessions/sessions.json`
  - 会话历史：`~/.openclaw/agents/<agentId>/sessions/<SessionId>.jsonl`

### 3. 记忆系统

- **双重记忆层**：
  - `memory/YYYY-MM-DD.md`：每日日志（仅追加）
  - `MEMORY.md`：精选长期记忆（仅在主私密会话中加载）
- **自动记忆刷新**：会话接近自动压缩时，触发静默的智能体轮次提醒模型写入持久记忆
- **向量记忆搜索**：
  - 默认启用，在 `MEMORY.md` 和 `memory/*.md` 上构建向量索引
  - 支持混合搜索（BM25 + 向量）、MMR 重排序（多样性）、时间衰减（近期提升）
  - 支持多种嵌入提供商：OpenAI、Gemini、Voyage、Mistral、Ollama、本地
- **工具**：
  - `memory_search`：语义搜索索引片段
  - `memory_get`：目标读取特定 Markdown 文件/行范围

### 4. 消息流程

1. **入站消息** → 路由/绑定 → 会话键
2. **队列**（如果有运行中的运行）
3. **智能体运行**（流式 + 工具）
4. **出站回复**（通道限制 + 分块）

- **去重**：短生命周期缓存键避免重复传递触发再次运行
- **去抖动**：同一发送者的快速连续消息可批量处理为单个智能体轮次
- **流式传输 + 分块**：
  - **块流式传输**：以粗块发送助手输出
  - **预览流式传输**：在生成时更新临时预览消息

### 5. 模型选择与故障转移

- **选择顺序**：主模型 → 回退模型
- **身份验证配置文件轮换**：同一提供商内轮换多个身份验证配置文件
- **冷却机制**：
  - 速率限制/超时错误使用指数退避（1分钟→5分钟→25分钟→1小时）
  - 计费/信用失败标记为禁用，使用更长的退避（5小时→10小时→24小时）
- **模型回退**：所有提供商配置文件失败时，移动到 `agents.defaults.model.fallbacks` 中的下一个模型

### 6. 队列系统

- **车道感知 FIFO 队列**：每个车道使用可配置的并发上限排水
- **按会话键排队**：保证每个会话只有一个活动运行
- **全局车道**：总体并行性受 `agents.defaults.maxConcurrent` 限制
- **队列模式**：
  - `steer`：立即注入当前运行
  - `followup`：排队等待当前运行结束后的下一个智能体轮次
  - `collect`：将所有排队消息合并为单个后续轮次（默认）
  - `steer-backlog`：立即转向并保留消息用于后续轮次
  - `interrupt`：中止该会话的活动运行，然后运行最新消息

### 7. 工具与插件

- **会话工具**：
  - `sessions_list`：列出会话
  - `sessions_history`：获取一个会话的记录
  - `sessions_send`：向另一个会话发送消息
  - `sessions_spawn`：在隔离会话中生成子智能体运行
- **插件钩子**：多个扩展点拦截智能体/工具生命周期和网关管道
  - `before_model_resolve`、`before_prompt_build`、`agent_end` 等

## 四、关键技术决策

1. **TypeBox 作为协议真实来源**：单一来源驱动运行时验证、JSON Schema 导出和 Swift 代码生成
2. **Markdown 格式化管道**：转换为共享中间表示（IR），然后渲染通道特定输出
3. **存在系统**：轻量级、尽力而为的网关和连接客户端视图，5分钟 TTL，最多 200 条目
4. **打字指示器**：可配置何时开始打字（never、instant、thinking、message）
5. **重试策略**：每个 HTTP 请求重试，而不是多步骤流程，保留排序
6. **使用跟踪**：直接从提供商使用端点拉取，没有估计成本

这个架构设计强调了一致性、可扩展性和用户体验的平衡，同时提供了强大的隔离和安全功能。
